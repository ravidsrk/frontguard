# Adversarial Review: Frontguard v0.2.0 Post-Ship Dossier

## Executive Summary

Frontguard v0.2.0 was tagged, packages were published to npm, and a launch banner went live — but the artifact does not survive adversarial inspection. Of 161 candidate findings, 49 (30%) survived double-lens verification (code reality + customer impact); 112 (70%) were refuted. The surviving picture is blunt: **the documented Quick Start fails on a clean machine** (init writes a `.ts` config the CLI cannot load); **the cloud API cannot detect regressions** (Daytona sandboxes never receive prior baselines, so every run returns `new_baseline`); **the dashboard session secret has a hardcoded production fallback** shipped in published npm code (any reader can mint a cookie for any user); **the marketed Docker renderer image is not published** (`frontguard/render:latest` is 404 on Docker Hub); **the headline pricing CTA** points at `app.frontguard.dev` which is NXDOMAIN; **the Pro tier advertises a feature** (`productionMonitoring`) whose own enforcement code returns HTTP 402 to Pro customers; **the GitHub App bootstrap PR pins to `@v1`** which does not exist; **the Slack app misreads cloud-api results** (every run reports "No visual regressions" regardless of outcome); **Storybook's headline play()-aware capture is silently best-effort** because `page.evaluate(string, arg)` does not invoke the function on the Playwright side; **every `npx frontguard …` snippet in the docs** invokes a non-existent npm package (the bin is `frontguard`, the package is `@frontguard/cli`); and the "0.0% pixel-only false positives on 43 recheck routes" launch-gate number is built on a methodology where every diff entry has `hasBaselineImage: false` (the byte-compare fast path short-circuits before any real diff runs). The 0.2.0 launch is not "code complete, awaiting distribution"; it is a launch in which the documented happy path, the marketed differentiators, and the headline credibility metric all fail their own evidence under direct verification.

## P0 — Ship-blocking findings

### Install path (P0)

### [install-1] `frontguard init` generates a .ts config that `doctor` and `loadConfig` cannot read (no TS loader registered)  (severity: P0, dimension: install-path)
**Summary.** On a brand-new project the documented quick-start is `init` → `doctor` → `run`. `init` writes `frontguard.config.ts` by default, but the shipped CLI uses a plain Node `import()` to load it — no `tsx`/`jiti` hook is registered, so doctor blows up with: `config file is invalid: Unknown file extension ".ts"`. The first thing a new user runs is the first thing that fails.
**Evidence.** Reproduced in /tmp/fg-test: `npm install @frontguard/cli` (0.2.0 from npm) → `npx frontguard init` writes frontguard.config.ts → `npx frontguard doctor` prints `❌ Configuration: config file is invalid: Unknown file extension ".ts" for /private/tmp/fg-test/frontguard.config.ts`. Loader code: `packages/cli/src/core/config.ts:337-341` uses `await import(fileUrl)` with no TS hook. The published bin `dist/cli/index.js` is a plain Node script (`#!/usr/bin/env node`, no `--import tsx`). `tsx` is listed as a runtime dep but never actually wired in.
**Recommendation.** Either (a) default `init --format=js` so the generated config is loadable out of the box, or (b) register the bundled `tsx` loader (`import { register } from 'tsx/esm/api'`) at CLI startup so `.ts` configs actually work. Whichever you do, doctor and `loadConfig` must agree.
**Verifier verdicts.** Lens A (code reality): fail — `init.ts:158` defaults `format` to `'ts'`; `config.ts:337-341` is a bare `import(fileUrl)`; `tsx@^4.21.0` sits in dependencies as dead weight. Lens B (customer impact): fail — failure lands on documented commands #2 and #3 of the README Quick Start on a launch-day release.

### [install-2] First-run baseline init explodes when node_modules is in the worktree — init's .gitignore additions are missing `node_modules/`  (severity: P0, dimension: install-path)
**Summary.** On a clean, committed repo (`git init && npm install && frontguard init && git commit -am init`) the very first `frontguard run` against a reachable URL fails 100% of comparisons with `git rm failed: spawnSync git ENOBUFS` because the orphan-baselines checkout drags 16,538 node_modules files into the worktree. `init` writes a `.gitignore` that explicitly omits `node_modules/`, so the user's repo is set up exactly the way it should not be for the orphan-branch strategy.
**Evidence.** Reproduced in /tmp/fg-test with `npx serve` on :8765, committed state. Log: `Preparing worktree (detached HEAD ...) Updating files: 100% (16538/16538)... Baseline storage init failed: git rm failed: spawnSync git ENOBUFS. All screenshots will be treated as new.` followed by 3 of 3 `Comparison failed: GitOrphanStorage not initialized.` lines. Report shows `Route unknown ✘ 3 errors`. Source: `packages/cli/src/cli/init.ts:247` hardcodes `entriesToAdd = ['auth.json', '.frontguard/', '.frontguard-debug/', 'frontguard-report/']` — no `node_modules/`.
**Recommendation.** Add `node_modules/` to the entries `init` appends to `.gitignore` (or, more robustly, have GitOrphanStorage detect a worktree that contains a `package.json` and refuse to checkout unless the working tree is clean of ignorable dirs — and emit a real error message, not "All screenshots will be treated as new" followed by an internal `not initialized` panic).
**Verifier verdicts.** Lens A (code reality): fail — `init.ts:247` literally omits `node_modules/`; `child_process.execFileSync` has no `maxBuffer` override in `git-orphan.ts:88-92`. Lens B (customer impact): fail — 100% failure on first run for any user who follows the natural `git init && npm install && frontguard init` ordering, with a cryptic two-line error that names neither node_modules nor the fix.

### [install-4] `--docker` references `frontguard/render:latest` which is not published to Docker Hub  (severity: P0, dimension: install-path)
**Summary.** `frontguard run --docker` re-execs the CLI inside `frontguard/render:latest` (see `DEFAULT_IMAGE` in `packages/cli/src/render/docker.ts:42`). That image is not on Docker Hub. The README explicitly markets "Cross-OS byte-equivalent baselines — pinned Docker renderer" as a key differentiator. A user who actually tries the docker mode will hit `pull access denied for frontguard/render` from Docker Hub the first time they invoke it.
**Evidence.** `curl https://hub.docker.com/v2/repositories/frontguard/render/` returns `{"message":"object not found"}`. Source: `packages/cli/src/render/docker.ts:42` hardcodes `frontguard/render:latest`; `packages/cli/src/cli/index.ts:265` advertises `--docker` flag with no warning about needing to build locally.
**Recommendation.** Either publish `frontguard/render:latest` (and version-pinned tags like `:v0.2.0` matching the Dockerfile at `packages/cli/docker/Dockerfile:37`) to Docker Hub, or pre-flight check `docker manifest inspect` and surface a `--docker-image-build-instructions` style error pointing the user at `packages/cli/docker/Dockerfile` so they can build it themselves.
**Verifier verdicts.** Lens A (code reality): fail — image is unambiguously not published (Docker Hub returns 404 and the `frontguard` user does not exist on Hub); `runDocker()` only preflights `docker --version`, never the image. Lens B (customer impact): fail — README/cross-os-rendering.mdx routes the exact target customer (mixed-OS teams) into the marketed `--docker` path, which 404s on first pull.

### [claim-5] Pro tier ($29/mo) advertises "Production monitoring scheduler" but the plan limits flag has it `false` — feature is Business-tier only  (severity: P0, dimension: marketing-claims)
**Summary.** The Pricing page Pro tier (`apps/landing/src/routes/pricing.tsx:71`) lists "Production monitoring scheduler" as an included feature. The actual gating in `packages/cloud-api/src/billing/plans.ts` has `pro.limits.productionMonitoring: false` (line 59) and only `business.limits.productionMonitoring: true` (line 73). So a customer who pays $29/mo for Pro would not actually get the headline feature in the included-feature list — `hasFeature(pro, 'productionMonitoring')` returns `false`. This is a textbook false-advertising bug, doubly bad because the page also presents a "Compare plans" matrix two sections lower (`MATRIX` at line 115) that shows the same scheduler as a Pro check.
**Evidence.** `apps/landing/src/routes/pricing.tsx:71` — Pro features list includes `'Production monitoring scheduler'`. `packages/cloud-api/src/billing/plans.ts:48-61` — Pro plan limits: `productionMonitoring: false`. `packages/cloud-api/src/billing/plans.ts:63-76` — Business plan limits: `productionMonitoring: true`. `packages/cloud-api/src/billing/plans.ts:107-110` — `hasFeature(plan, 'productionMonitoring')` reads exactly that boolean. The runtime enforces this: `packages/cloud-api/src/routes/monitors.ts:68` returns HTTP 402 with `"Production monitoring is not available on the Pro plan."` to any paying Pro customer who tries to create a monitor.
**Recommendation.** Decide intent — either flip `pro.limits.productionMonitoring` to `true` (and update the cron route to allow the Pro plan), or remove the feature from the Pro tier's `features` array and from the `MATRIX` row. **Verifier verdicts.** Lens A (code reality): fail — pricing page advertises the feature in two places (bullet list + matrix); runtime returns HTTP 402 with the exact phrase "not available on the Pro plan." Lens B (customer impact): fail — 402 on a headline paid feature is a refund request and a social-grade complaint; the customer cannot self-serve to Business because Business is not on the pricing page.

### Cloud API (P0)

### [cloud-1] Worker can never detect regressions — Daytona sandbox is ephemeral and prior baselines are never uploaded  (severity: P0, dimension: cloud-api)
**Summary.** Every cloud run spins up a fresh Daytona sandbox via `daytona.create({ ephemeral: true })` and only uploads `frontguard.config.json`. The previous run's baseline PNGs (which live in R2 under the user's prefix) are never restored to the new sandbox before `frontguard run` executes, so each invocation produces baseline-marker results. The processor's own no-Daytona branch already documents this: it returns `status: 'new_baseline'` for every route/viewport. The Daytona branch has the same bug because there's no R2-to-sandbox sync. Result: POST /v1/run can take screenshots but can never raise a regression — the headline product feature is non-functional on the cloud path.
**Evidence.** `packages/cloud-api/src/daytona-runner.ts:64-77` creates an ephemeral sandbox; lines 102-148 only upload `frontguard.config.json` (line 135) before running `frontguard run`. Grep `uploadFile|baseline` shows the only uploaded file is the config. R2 keys for prior baselines (`storage/screenshots.ts:49-59`) exist but no code reads them back into the sandbox. `processor.ts:84-93` confirms the alternate branch returns `status:'new_baseline'` for every route.
**Recommendation.** Before invoking `frontguard run` in the sandbox, the runner must (1) list R2 objects under `<userId>/<projectId-or-run-key>/baseline/*`, (2) download them to the sandbox baseline directory, and (3) only then exec. Without this, the entire cloud product is a "screenshot service", not a regression detector. This must block the v0.2.0 launch claim.
**Verifier verdicts.** Lens A (code reality): fail — `daytona-runner.ts` has exactly one `sandbox.fs.uploadFile` call (the config); zero R2-to-sandbox restore code path exists. Lens B (customer impact): fail — `baselinesApproved` flag, dashboard, Slack/email alerts, team review workflow, and every paid feature in the cloud all depend on a regression signal that is structurally unreachable.

### [cloud-4] Dashboard session secret silently falls back to a hardcoded string published in source  (severity: P0, dimension: cloud-api)
**Summary.** `auth/session.ts:29` exports `DEV_SESSION_SECRET = 'frontguard-dev-session-secret'` and `sessionSecret()` uses `env?.DASHBOARD_SESSION_SECRET || DEV_SESSION_SECRET`. If the operator forgets to set `DASHBOARD_SESSION_SECRET` (it is listed in `wrangler.toml` as a documented secret but is NOT enforced anywhere), every dashboard cookie is signed with a public constant baked into npm-published code. Any attacker who reads the OSS repo can mint a valid `fg_session=<victimUserId>.<expiry>.<HMAC>` cookie and impersonate any user on the dashboard. There is no startup check that the secret is set in production.
**Evidence.** `src/auth/session.ts:29: export const DEV_SESSION_SECRET = 'frontguard-dev-session-secret';`. `src/auth/session.ts:32-34`: silent fallback. `wrangler.toml:39-48` enumerate Stripe / Daytona / GitHub secrets but never mention `DASHBOARD_SESSION_SECRET`, so an operator following the comment will simply not set it. No assert/throw if `env.DASHBOARD_SESSION_SECRET` is unset while `env.DB` is set (production).
**Recommendation.** When `isProduction(env)` is true and `env.DASHBOARD_SESSION_SECRET` is undefined, refuse to mint any session cookie and return 503 with "Dashboard not configured". Better yet, make `DASHBOARD_SESSION_SECRET` a required boot-time check in `index.ts` that throws on first request when missing in prod. Rotate the `DEV_SESSION_SECRET` constant out of the published package or rename it `INSECURE_DEV_SESSION_SECRET_DO_NOT_USE_IN_PROD`.
**Verifier verdicts.** Lens A (code reality): fail — constant ships in `dist/index.js:1498`; `sessionSecret()` has no `isProduction` guard; wrangler.toml documented-secrets list omits it. Lens B (customer impact): fail — pre-auth account takeover of every dashboard user via a documentation-only mitigation that is itself missing.

### Security & auth (P0)

### [sec-1] Dashboard session secret has a hardcoded production fallback — anyone reading the source can forge any user's session cookie  (severity: P0, dimension: security-auth)
**Summary.** `auth/session.ts` exports `DEV_SESSION_SECRET = 'frontguard-dev-session-secret'` and `sessionSecret()` silently falls back to it when `DASHBOARD_SESSION_SECRET` is not set. The cloud-api wrangler.toml lists every other secret to `wrangler secret put` (`GITHUB_CLIENT_SECRET`, `DAYTONA_API_KEY`, `STRIPE_*`, etc.) but omits `DASHBOARD_SESSION_SECRET`, so a real deploy can easily forget it. Forging is trivial: `HMAC-SHA256(<userId>.<expiry>, 'frontguard-dev-session-secret')`.
**Evidence.** `packages/cloud-api/src/auth/session.ts:29 export const DEV_SESSION_SECRET = 'frontguard-dev-session-secret';` and line 33 `return env?.DASHBOARD_SESSION_SECRET || DEV_SESSION_SECRET;`. `wrangler.toml:39-49` documents every other secret but no `DASHBOARD_SESSION_SECRET`. The constant is also shipped in `packages/cloud-api/dist/index.js:1498`. Forged a working cookie locally: `attacker-forged-user.1782293265.FKaFPlL49b4VlcOnM_dP2-UEm6cuvo4AZKufFxeHO7M`. Whoever holds that cookie reads `/dashboard/*` for any user, lists runs, downloads R2 attachments via `/dashboard/runs/:id/attachments/:id/download`, etc.
**Recommendation.** Fail closed: require `DASHBOARD_SESSION_SECRET` in production (`isProduction(env)`) and throw at startup if missing. Delete the `DEV_SESSION_SECRET` constant or move it behind a `process.env.NODE_ENV !== 'production'` runtime check. Add `DASHBOARD_SESSION_SECRET` to wrangler.toml's documented secrets and to `docs/launch-readiness.md`.
**Verifier verdicts.** Lens A (code reality): fail — forgery reproduced in 4 lines of Node; 24+ dashboard routes accept the forged cookie. Lens B (customer impact): fail — single global HMAC key on public GitHub means account takeover of every customer with one curl; SOC2/GDPR mandatory-disclosure event.

### Integrations depth (P0)

### [int-1] Slack app reads a fabricated run-result shape — every message will report "No visual regressions"  (severity: P0, dimension: integrations-depth)
**Summary.** `integrations/slack-app/src/runs.ts:45` declares `CloudRunStatus.results` as `Array<{regression?: boolean; warning?: boolean}>`, but the real cloud-api emits `RunResult` with `status: 'passed'|'regression'|'changed'|'warning'|'new_baseline'|'failed'` (see `packages/cloud-api/src/types.ts:30-37` and `processor.ts:53-60`). `summarizeRun()` at `runs.ts:128-144` iterates and checks `r.regression` and `r.warning`, which never exist on the wire payload. The unit test (`test/runs.test.ts:178-202`) only passes because it hand-feeds the wrong shape — there is no test against the real cloud-api shape, unlike `netlify/test/core.test.js:177-200` which correctly tests the real shape.
**Evidence.** `integrations/slack-app/src/runs.ts:45 results?: Array<{ regression?: boolean; warning?: boolean }>`; `runs.ts:133-135 for (const r of results) { if (r.regression) regressions++; else if (r.warning) warnings++; }`; vs `packages/cloud-api/src/types.ts:30-37 RunResult { status: string; … }`; `processor.ts:53-60` emits `status: r.status`. Netlify (`integrations/netlify/lib/core.js:215-227`) reads `r.status` correctly.
**Recommendation.** Replace `summarizeRun()` with the same status-string logic netlify uses (`status === 'regression' || 'changed' || 'failed'` etc.) and add a fixture-driven test parallel to `netlify/test/core.test.js`'s `FIXTURE_FAILING`.
**Verifier verdicts.** Lens A (code reality): fail — `GET /v1/runs/:id` returns the raw shape; Slack's iterator counts 0 regressions on every actual response; tests hand-feed a shape the API never emits. Lens B (customer impact): fail — flagship v0.2 integration silently inverts the product's core signal on 100% of completed runs.

### [int-3] GitHub App bootstrap PR pins to `ravidsrk/frontguard@v1` — a ref that does not exist  (severity: P0, dimension: integrations-depth)
**Summary.** On every fresh install the app opens a bootstrap PR that adds `.github/workflows/frontguard.yml` pinning to `uses: ravidsrk/frontguard@v1` (`integrations/github-app/src/github-api.ts:362 ACTION_REF = 'ravidsrk/frontguard@v1'`). The repo has only `v0.1.0` and `v0.2.0` tags — no `v1`. Worse, GitHub Actions resolves `<owner>/<repo>` against the repository root's `action.yml`, but the action.yml lives at `packages/cli/action.yml` — there is no action.yml at the repo root. So even if a v1 tag existed, every CI run from a bootstrap PR would fail with "Unable to resolve action ravidsrk/frontguard@v1, action.yml not found." Every brand-new install will get a red workflow on day one.
**Evidence.** `integrations/github-app/src/github-api.ts:362 export const ACTION_REF = 'ravidsrk/frontguard@v1';`. `DEFAULT_WORKFLOW_YML` (lines 373-391) interpolates `ACTION_REF` into the `uses:` line. `git tag --list` → only v0.1.0, v0.2.0. `find . -name action.yml -not -path '*/node_modules/*'` → only `packages/cli/action.yml` (sub-path; GH resolves repo-root only).
**Recommendation.** Either tag the repo `v1` (lightweight tag pointing at a stable commit AND move/copy `action.yml` to the repo root, since GitHub does not honour sub-path resolution by default), or change `ACTION_REF` to a sub-path form `ravidsrk/frontguard/packages/cli@v0.2.0` (supported by composite/Docker actions only — verify your action.yml shape). Also add a CI smoke test that calls `actions/checkout@v4` then `uses: ravidsrk/frontguard@<ACTION_REF>` against the local repo so this drift cannot ship again.
**Verifier verdicts.** Lens A (code reality): fail — no v1 tag, no repo-root action.yml; integrations/github-app/README.md:19 reinforces the broken ref. Lens B (customer impact): fail — every new install gets a Day-1 red CI run on the PR Frontguard itself opened.

**Verifier verdicts.** Lens A (code reality): fail — confirmed against built and dev-served Storybook 8.6.18; zero `parameters`/`frontguard` matches across 7 stories; the unit tests hand-fabricate a shape production never produces. Lens B (customer impact): fail — `frontguard init` and docs both route the customer toward `parameters.frontguard` as the per-story knob; all four worked examples in storybook.mdx silently no-op.

### [sb-2] Documented fixture command `npx frontguard run --config ./frontguard.config.ts` fails — CLI cannot load a TypeScript config  (severity: P0, dimension: storybook)
**Summary.** Both the integration docs (`storybook.mdx:495 npx frontguard run --config ./frontguard.config.ts`) and the fixture README explicitly tell users to run with the TS config. **Verifier verdicts.** Lens A (code reality): fail — reproduced verbatim with the published artifact; auto-discovery also tries `frontguard.config.ts` first. Lens B (customer impact): fail — `frontguard init` defaults to `.ts` and the README Quick Start runs `frontguard run` next; every TS-using customer following the Quick Start hits this on command #2.

### [sb-3] STORYBOOK_READY_SCRIPT ready-wait throws on every story — play()-await is silently best-effort, not guaranteed  (severity: P0, dimension: storybook)
**Summary.** The renderer logs `Storybook ready-wait failed for /iframe.html?id=...: Cannot read properties of undefined (reading 'reason')` for every single story when run against a real Storybook 8.6 preview. The catch on `packages/cli/src/render/playwright.ts:281-285` swallows it. The error means `result` from `page.evaluate(STORYBOOK_READY_SCRIPT, sbTimeout)` is `undefined`, so the documented behavior — "Polls `window.__STORYBOOK_PREVIEW__.storyRenders` for completed/played" (`storybook.mdx:269-275`) — never actually runs. ### Validation soundness (P2)

### [val-5] Two-pass recheck against unchanged code is not an independent measurement — `pixelmatch` short-circuits on byte-identical PNGs  (severity: P2, dimension: validation-soundness)
**Summary.** The recheck pass renders the same code at the same SHA in the same temp dir against the same orphan-branch baseline. `packages/cli/src/diff/pixel.ts:51-63` takes a fast path when `Buffer.compare(current, baseline) === 0` and returns `status:'pass', diffPercentage:0`. Modern headless Chromium PNG output is largely deterministic for static content, especially over a single render with no anti-flake (val-4). The 0% FP rate is, to first order, measuring 'does Chromium produce a byte-identical PNG twice'. That is not a regression-detection benchmark; it is a determinism test on Chromium's encoder.
**Evidence.** `packages/cli/src/diff/pixel.ts:51-63` — byte-identical fast path. `results-v0.2.md:46-49` — recheck "against the same unchanged code". val-4 — anti-flake disabled, so there's no sub-pixel font rendering / animation noise being averaged out. `validation/results/tailwind-dashboard.json:40-99` — every diff entry shows `diffPercentage: 0` (not, say, 0.001%), consistent with byte-identical-PNG short-circuit.
**Recommendation.** To actually measure FP rate, the recheck pass should boot a FRESH dev-server process (forcing fresh CSS / JS hashes / font sub-pixel decisions), or render against a separate clone, or seed the recheck with a no-op `git commit --allow-empty` and a forced server restart. The current setup is too friendly to the diff engine to be informative.
**Verifier verdicts.** Lens A (code reality): fail — fast path confirmed; harness shares same dev-server PID + WORK_DIR + SHA + baseline between passes; antiFlakeRenders defaults to 1. Lens B (severity adjusted P1→P2) — no customer-facing defect; methodologically thin marketing metric on a single launch document.

## Coverage gaps (completeness critic)

The 15 dimensions above caught 49 confirmed defects, but several attack surfaces a v0.2.0 hostile competitor or enterprise procurement reviewer would probe were never inspected. The items below are not findings — they are "we did not look here, and we should." Each one is something a buyer-side red-team or a press fact-check could weaponise within an hour.

### [gap-1] OSI / license-compliance audit of the dependency tree
**Why it matters.** The repo is MIT-licensed and the README markets "MIT-licensed" as a differentiator versus Chromatic ("proprietary"). But the audit never enumerated transitive license obligations. If the production tree contains a GPL, AGPL, BUSL, SSPL, or Elastic-2.0 dependency, the "MIT" badge is misleading at best, license-infringing at worst. `@daytonaio/sdk` is the highest-risk pivot — it ships under the `@daytonaio` org which has shifted licensing models for cloud sandboxes (Daytona shifted to AGPL for its core in 2024), and its transitive OpenTelemetry chain pulls in `protobufjs` (BSD-3) plus other licenses we never enumerated.
**How to audit.** Run `npx license-checker --production --excludePrivatePackages --summary` and `npx license-checker --production --failOn 'GPL-3.0;AGPL-3.0;SSPL-1.0;BUSL-1.1'` at the repo root and inside each published package. Run `npm view @daytonaio/sdk license` and chase its top-10 transitive deps. Generate an SBOM with `npx @cyclonedx/cyclonedx-npm --output-file sbom.json` and feed it into Fossa/ScanCode. Diff "what we shipped on npm" against "what we license as MIT" — every conflict is a P1.

### [gap-2] GDPR / data-subject-rights flow in the cloud-api
**Why it matters.** `docs/retention.md` was authored as part of v0.2.0 launch and presumably promises data-export, data-deletion, retention windows, and DPA terms. The audit never verified the cloud-api actually implements those promises. If retention.md says "30-day retention on screenshots" but R2 has no lifecycle policy, or if it says "data export endpoint" and there is no `GET /v1/users/me/export`, the gap is a regulatory mis-statement on a public docs page — every enterprise procurement form in the EU asks for this in writing.
**How to audit.** Read `docs/retention.md` line-by-line and grep the cloud-api source for each promise. Specifically: `grep -rnE 'lifecycle|expir|retention|delete|export' packages/cloud-api/src` and check that R2/D1 actually has the matching code. Verify `wrangler.toml` declares lifecycle rules on the R2 bucket. Check whether `/v1/users/me` supports `DELETE` (right-to-be-forgotten) and `GET ?format=archive` (data portability). Confirm screenshots reachable via signed URLs honour the documented TTL.

### [gap-3] Telemetry contract — what is sent home, is opt-out documented, is the endpoint live
**Why it matters.** Finding install-9 covered "the telemetry endpoint is NXDOMAIN" but never inspected *what* the CLI tries to send. If the payload includes file paths, URL targets (which can encode customer hostnames), AI prompts (which can encode UI strings), or any PII, the README's silent default-on telemetry is a credibility/disclosure failure independent of whether the endpoint resolves. Also `docs/telemetry.md` exists (70 lines) and was never reconciled against actual code.
**How to audit.** `grep -rnE 'telemetry|trackEvent|sendBeacon|posthog|segment|amplitude' packages/cli/src packages/cloud-api/src integrations/` and read the payload-construction site. Confirm fields against `docs/telemetry.md`'s declared schema. Run `frontguard run --update-baselines` with `tcpdump`/`mitmproxy` and capture the actual outbound request body. Verify `--no-telemetry`, `FRONTGUARD_TELEMETRY=0`, and `DO_NOT_TRACK=1` all suppress. Confirm telemetry is disabled in CI environments by default (test against `CI=true`).

### [gap-4] Accessibility (WCAG 2.2 AA) of the landing + dashboard + report HTML
**Why it matters.** Frontguard's pitch is "visual regression for the modern web" — selling visual quality is the whole product. Shipping a landing page or a customer-facing report that fails WCAG 2.2 AA on contrast, focus, ARIA, or keyboard nav is a hostile-press headline ("Visual-QA company can't pass visual-QA"). Also, US Section 508 and EU EAA (effective 2025-06-28) are now enforceable for any SaaS sold into government / EU consumers — Frontguard's pricing page CTA is "Start trial" which makes it consumer-facing in EAA's reading.
**How to audit.** Run `axe-core` or `pa11y` against `https://frontguard.dev`, `https://frontguard.dev/pricing`, `https://docs.frontguard.dev`, and a sample cloud-api `/v1/reports/:id` HTML report. Confirm the dashboard at `app.frontguard.dev` (once DNS lands) passes — but in the meantime audit the source in `packages/cloud-api/src/dashboard*.ts` and `apps/landing/src/`. Verify color contrast on the brand palette (`branding/`), keyboard-only navigation of the comparison/diff approval UI, and that AI-suggested fix descriptions are properly announced to screen readers.

### [gap-5] Cross-OS rendering correctness beyond byte-equivalence (font hinting, sub-pixel AA)
**Why it matters.** Finding docker-3 caught the `--platform` pin gap, but only at the architecture layer. Even with `linux/amd64` pinned, font rendering depends on the bundled FreeType/HarfBuzz versions, the system fontconfig cache, the presence of specific font families (Helvetica vs Liberation Sans fallbacks), and locale-driven Unicode line-break/shaping rules. A customer whose site uses a non-Latin script (Hindi, Arabic, CJK) will get different baselines on Docker-render vs cloud-render even with byte-identical bytes everywhere upstream. The README markets "Cross-OS byte-equivalent" without script/locale qualifications.
**How to audit.** Render the same complex-script test page (e.g., Devanagari, Arabic RTL, CJK with vertical writing-mode) under (a) Docker on macOS amd64-emulated, (b) Docker on Linux amd64-native, (c) cloud Daytona sandbox, (d) bare-metal CI runner. Pixel-diff every pair. Confirm fontconfig cache is deterministic across container layers. Verify the Dockerfile pins specific font packages (`fonts-noto-cjk`, `fonts-noto-color-emoji`, etc.) and that the Playwright base image does too. Confirm `LANG` / `LC_ALL` are explicitly set in `runDocker`.

### [gap-6] Storage cost & R2 lifecycle policy on cloud-api
**Why it matters.** Every run uploads N screenshots × M viewports × B browsers to R2 under the user's prefix. The pricing page sets the Pro tier at $29/mo. If R2 has no lifecycle policy and a Pro customer runs 10 PRs/day × 50 routes × 3 viewports × 90-day retention, they generate ~1.35M PNGs/quarter. Class B operations + storage = COGS that may exceed $29/mo, especially since the cloud-api also fronts `/dashboard/runs/:id/attachments/:id/download` with no per-key rate limit (gap candidate; not audited). A hostile competitor would publish "Frontguard sells at a loss" with the cost math.
**How to audit.** Read `wrangler.toml` and verify R2 bucket lifecycle rules. Check `packages/cloud-api/src/storage/screenshots.ts` for an upload-side TTL. Model COGS at the Pro tier's free quota with the cloud-api's documented limits and confirm gross margin is positive at 1×, 10×, 50× usage. Verify that signed-URL TTL on attachments is short enough to prevent hot-linking by abusive consumers.

### [gap-7] Disaster recovery — D1 backup schedule, R2 redundancy, secret rotation
**Why it matters.** The cloud-api stores users, runs, baselines (the customer's source of regression-detection truth) in Cloudflare D1 and R2. Neither service has automatic point-in-time recovery enabled by default — D1 requires `wrangler d1 backup create` to be scheduled. If a `DROP TABLE runs` slips through a migration (or an Anthropic-issued `DELETE FROM runs WHERE 1=1` via a future MCP tool), there is no recovery story. Enterprise SOC2 Type II requires documented backup, restore-test, and RTO/RPO numbers. The dossier never inspected any of this.
**How to audit.** `grep -rn 'd1 backup\|backupSchedule\|export' .github/ scripts/ packages/cloud-api/` for any scheduled backup. Verify the `wrangler.toml` has a `[backups]` block or a Cron Trigger that invokes `wrangler d1 export`. Verify R2 has `Object Versioning` enabled (Cloudflare R2 supports it as of 2024-Q4). Confirm `DASHBOARD_SESSION_SECRET`, `STRIPE_*`, `GITHUB_*` secrets have a documented rotation procedure (not just an initial set instruction). Check that migrations are reversible (`packages/cloud-api/src/db/migrations/` should have `down` SQL).

### [gap-8] Bus factor / long-term maintenance signal
**Why it matters.** `git log --format='%aN' | sort -u` and the GitHub repo's contributor list almost certainly show 1 person (Ravindra Kumar). Enterprise procurement asks "what happens if your lead engineer is hit by a bus" — this is a real survey question on SOC2 questionnaires and on every >$50k vendor form. Frontguard markets itself as a Chromatic/Percy replacement (paid SaaS, multi-year customer commitments) but does not address the bus factor in any docs/* file we audited. A competitor's marketing team would publish "single-developer project, no SLA, will not exist in 2027" against this surface.
**How to audit.** Count unique committers in `git log` for the last 90/180/365 days. Audit `CONTRIBUTING.md` for an actual review-by-second-engineer policy. Check whether GitHub branch protection requires `>=1 approving review from CODEOWNERS` (and whether CODEOWNERS exists). Verify there is a documented runbook for "Ravindra is unreachable for 30 days." Check whether the npm package's 2FA + maintainer count is >=2 (`npm owner ls @frontguard/cli`).

### [gap-9] Stripe billing correctness — webhook signature verification, idempotency, refund flow
**Why it matters.** Finding claim-5 caught that the Pro tier advertises a Business-tier feature, but the audit never inspected the billing-side code path: Stripe webhook signature verification (replay attacks), idempotency on plan upgrades/downgrades, proration on mid-cycle changes, refund handling for finding claim-5's customer who gets HTTP 402 on Pro. If `packages/cloud-api/src/billing/` doesn't verify `Stripe-Signature`, an attacker can POST a forged `customer.subscription.updated` event and self-promote to Business tier. If there's no idempotency key on `invoice.payment_succeeded`, a network retry can double-grant credits.
**How to audit.** Read `packages/cloud-api/src/billing/webhook*.ts` (if it exists) and verify it calls `stripe.webhooks.constructEvent` with the raw body and the signature header. Confirm `STRIPE_WEBHOOK_SECRET` is required at runtime (no fallback like sec-1). Verify event IDs are stored in D1 to prevent replay. Test the refund / cancellation flow end-to-end: does `customer.subscription.deleted` actually revoke API key access, or do already-issued keys keep working for 30 days?

### [gap-10] Rate limiting & DoS resistance on cloud-api
**Why it matters.** `POST /v1/run` is the most expensive endpoint in the product — it spins up a Daytona sandbox per invocation. A single malicious or buggy customer running `while true; do curl -X POST /v1/run -d '{...}'; done` can rack up unbounded Daytona costs against Frontguard's account. The audit never inspected per-key rate limits, concurrent-run caps, sandbox-cost ceilings, or circuit breakers. Cloudflare Workers' built-in rate limiting is not free / not on by default.
**How to audit.** `grep -rnE 'rateLimit|throttle|RateLimit|cf-rate-limit' packages/cloud-api/src` and verify presence. Check whether `wrangler.toml` enables `[[unsafe.bindings]]` for `rate_limiter`. Inspect billing/plans.ts limits (`runsPerMonth`, `concurrentRuns`) and confirm they're actually enforced before sandbox creation, not after. Verify `daytona-runner.ts` has a per-customer cost budget / sandbox-lifetime cap. Test: submit 100 parallel `/v1/run` requests with one API key and confirm rejection on number 11 (or whatever the documented limit is).

### [gap-11] Sandbox escape / sandbox abuse — what can a hostile `frontguard.config.json` do inside Daytona?
**Why it matters.** Finding cloud-1 caught that baselines aren't restored to the sandbox, but the inverse question wasn't asked: what can the sandbox *do* on behalf of the user? `frontguard.config.json` (or `.ts`) accepts arbitrary URLs, custom Playwright `goto` options, before-each scripts (per the docs), and the `--ai` flag invokes a remote LLM with prompts derived from screenshots + page HTML. A malicious customer could submit a config that scrapes Frontguard's other Daytona snapshots, exfiltrates environment variables, or exhausts the sandbox's egress quota. The audit never inspected Daytona sandbox network policy, mount restrictions, or env-var scoping.
**How to audit.** Read `scripts/build-daytona-snapshot.ts` and confirm the snapshot has no Frontguard secrets baked in. Verify Daytona's network egress policy blocks `169.254.*`, `metadata.google.internal`, `*.frontguard.dev`. Check whether `runRequestSchema` strips `before`/`after` lifecycle hooks from user-submitted configs. Confirm AI vision prompts are bounded in token count and length. Verify R2 access from sandbox is scoped to the requesting user's prefix only (or non-existent — the sandbox shouldn't talk to R2 directly).

### [gap-12] Internationalization of the renderer + UI strings
**Why it matters.** The renderer's `STORYBOOK_READY_SCRIPT`, the AI vision prompts, the report HTML, the dashboard, and the error messages all assume English. A customer in Tokyo or São Paulo who installs Frontguard gets an English-only product. More importantly, the renderer's behaviour can diverge for non-Latin scripts — `page.goto` with `waitUntil:'networkidle'` interacts differently with Japanese IME injection, RTL Arabic layouts, and CJK font-loading. The audit never probed any of this.
**How to audit.** `grep -rnE 'lang=|locale|i18n|intl' apps/docs apps/landing packages/cli/src/render` and inventory locale assumptions. Render a known RTL page (e.g., Wikipedia Arabic) under `--docker` and compare against bare-metal output. Verify date/time strings in the report HTML respect `Accept-Language` or are at least timezone-explicit (UTC). Confirm `--locale` and `--timezone-id` flags exist on `frontguard run` and propagate to Playwright's `browser.newContext({locale, timezoneId})`.

### [gap-13] Cross-browser rendering parity — Firefox + WebKit beyond Chromium
**Why it matters.** The validation harness ran 43 routes through Chromium-only. The cloud-api `RunRequest` accepts `browsers: ['chromium', 'firefox', 'webkit']` and the CLI docs market multi-browser. But finding mcp-9 already noted that the cloud-api throws away the `browser` field on result emission, and the audit never confirmed that Firefox or WebKit baselines are actually captured, stored separately, and diffed independently in real runs. If only Chromium works in practice, the "test in Firefox + Safari" pitch on the comparison table is empty.
**How to audit.** Submit a real `/v1/run` with `browsers: ['firefox', 'webkit']` against a public test URL and inspect the resulting R2 prefixes — confirm three separate baseline directories. Open the report HTML and verify three separate diff cells per route. Verify `packages/cli/src/render/playwright.ts` actually exercises the firefox/webkit codepath (`grep -nE 'firefox|webkit' packages/cli/src/render/`). Confirm Storybook play()-aware capture works on Firefox + WebKit (sb-3 only reproduced on Chromium).

### [gap-14] Trademark / brand-name conflict
**Why it matters.** A hostile competitor's marketing team's first move is "does this name conflict with anyone's mark?" The name "Frontguard" has no audit trail in the dossier for prior-art / trademark conflict. If there's an existing "FrontGuard" or "Front Guard" trademark in software, security, or even adjacent categories (US class 9 / 42 cover both), the project faces a cease-and-desist on the same day it gets press coverage. Domain `frontguard.dev` doesn't prove trademark availability — the `.dev` TLD is registrar-policy land, not USPTO.
**How to audit.** Search USPTO TESS for `frontguard*` and `front guard*` (live + dead, classes 9 / 35 / 42 / 45). Search EUIPO eSearch for the same. Check google's "FrontGuard" results outside the developer-tools space (we did not). Verify the npm `@frontguard` org is owned by Ravindra and not by another party. Check whether there are prior products called "Frontguard" in security/QA/observability where a customer could confuse the two.

### [gap-15] What a hostile competitor's marketing team would attack v0.2.0 from — the "unknown unknown"
**Why it matters.** The 15 reviewers worked from the project's own docs and code. They did not simulate "Chromatic's marketing lead writes a blog post titled 'Five reasons not to switch to Frontguard'." The strongest such attack is not any single P0 — it is the *compositional* attack: "Frontguard's launch shipped with (a) broken DNS, (b) a 404 npm package referenced in every doc, (c) zero ability to actually detect regressions in the cloud, (d) a hardcoded session secret on public GitHub, (e) two critical CVEs in the dep tree, (f) a launch-gate metric measured against a code path that short-circuits, and (g) a 0-star repo serving Schema.org 4.8/36 reviews — and they tagged this v0.2.0 and announced it." Time it. If the post can be written in <2 hours of effort from public evidence (no insider knowledge), the launch is exposed at the *narrative* layer, not just the defect layer — and that is the level that determines press coverage, HN front-page polarity, and procurement-team perception.

### Audit-thinness notes (where the existing 49 findings are themselves under-sampled)

- **Sample size on the install path was a single clean machine on macOS arm64.** Linux x86_64, Windows WSL2, and Node 20 vs 22 vs 24 were not separately probed. The npm tarball stale-README finding (install-7) and the bin-name finding (docs-1) likely have additional siblings on Windows (where `npx frontguard` resolves PATH differently and the `frontguard-render` sibling bin fails to install with execute permissions).
- **Storybook reproduction used Storybook 8.6.x only.** Storybook 9.0 (released 2025) changes the `/index.json` schema again; the sb-1 finding may be even worse there.
- **Daytona / cloud-api findings were verified against the source code, not against a live cloud-api deployment** (because `api.frontguard.dev` is NXDOMAIN). Several of the cloud-api findings are "from-source" verdicts that could shift if a live deployment exists somewhere else (e.g., a private Cloudflare Pages preview URL that wasn't in scope).
- **No reproduction commands were captured for the integrations** (Vercel, Netlify, GitHub App, Slack). The findings rely on code-reading; live end-to-end test against an actual Vercel/Netlify deploy was not performed because the marketplace listings are 404.
- **No browser-level testing of the dashboard.** The session-secret finding (sec-1) was code-only; a live forged-cookie session against a running dashboard worker was not exercised.
- **The validation 0.0% FP finding (val-5) was based on result JSON inspection.** No alternative measurement was attempted (fresh dev-server PID, multi-pass with restart, font sub-pixel forced re-render). A more rigorous critique would re-run the harness with byte-compare disabled to see what the FP rate actually is on the same 43 routes.
